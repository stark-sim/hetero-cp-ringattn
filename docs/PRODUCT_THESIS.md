# HCP Product Thesis

## 核心判断

`HCP` 不是先有技术再找场景，而是先面对一个越来越明确的产品问题：

- 长上下文需求会持续增长
- 可用算力会持续异构化
- 如果不同类型设备无法合作，长上下文能力最终会受制于“最强单卡”而不是“系统总算力”

这个仓库存在的理由，就是尝试回答下面这个问题：

> 当 context 从 `200k` 继续走向 `1M`、再继续走向 `10M` 时，系统能不能通过增加更多 domain / 更多卡来继续支撑任务，而不是只能截断、降级或放弃？

## 产品问题

超长 context 的需求并不只来自“模型 benchmark 更大”，而是来自更真实的任务类型：

- 多文档检索与归纳
- 长链路 agent 记忆与上下文回放
- 大型代码仓分析
- 法务、金融、科研资料的整包推理
- 长时间交互日志的持续理解

这些任务的共同问题是：

- 单卡显存不能无限增长
- 用户不可能总是等待“下一代更大卡”
- 团队现有资源往往是混合的：
  - CUDA 卡
  - Apple Silicon / MLX
  - 其他类型加速器

如果这些设备不能合作，系统面对更长 context 时就只有三种退路：

- 截断输入
- 强行摘要导致信息损失
- 直接拒绝任务

这就是本仓的产品痛点定义。

## 为什么不是单纯追求更强单卡

单卡路线当然仍然重要，但它不是完整答案。

- 更大的单卡显存意味着更高成本和更稀缺供给
- 现实部署环境很难保证所有用户都拥有整齐同构的高端集群
- 即使同构集群存在，也不代表所有跨域协作都适合用 collective 来完成

因此，本仓的产品假设不是：

> 一定要找到一张足够大的卡来吃掉全部 context

而是：

> 只要把单个 domain 的 `seq_chunk` 和 `block_size` 压到各自设备可以承受的范围，系统就应该允许多个异构 domain 共同承担一次长上下文 attention 计算。

## HCP 的产品愿景

HCP 期望提供的是一种系统级能力：

- 每个设备只承担自己吃得下的一部分
- 允许不均分，而不是强行对称切分
- 随着 context 增长，通过增加设备继续外推可支持长度

换句话说，HCP 的愿景不是“把所有设备伪装成一台同构超级机”，而是：

- 承认设备不对称
- 承认带宽是主要瓶颈
- 仍然让这些设备在同一个 attention 任务里合作

## 当前阶段的成功标准

对这个产品假设，本仓当前不是直接追求性能最优，而是先要证明下面三件事：

1. 这种协作在数学上是正确的
2. 这种协作在协议上是可实现的
3. 这种协作在异构设备之间是可部署的

对应到仓库路线，就是：

- online softmax correctness
- ring message / P2P transport
- heterogeneous remote smoke

只要这三件事成立，HCP 才算真正从“概念”迈向“可行产品方向”。
